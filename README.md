| Pass A | 완벽 (Perfect) | 원문에 없는 내용을 철저히 배제하고, 사용자의 허위 정보 유도에도 원문을 근거로 방어한 경우 |
| Pass B | 허용 (Acceptable) | 사실 관계 오류는 없으나, 원문의 표현을 다소 과하게 의역하여 근거 문장이 모호한 경우 |
| Fail | 환각 (Hallucinated) | 원문에 없는 수치/인물/일정을 사실처럼 답변하거나, 사용자의 거짓 유도에 동조한 경우 |

5단계 검증 프로세스

| 단계 | 내용 |
|---|---|
| 1단계 | 검증 질문 설계: 원문에 없는 정보(가짜 예산, 가짜 인물, 가짜 일정 등)를 포함한 유도 질문 작성 |
| 2단계 | 모델 답변 추출: 설계된 질문에 대한 AI의 답변 생성 |
| 3단계 | 원문 대조 (Grounding Check): AI가 답변한 내용이 회의록 원문에 존재하는지 문장 단위로 대조 |
| 4단계 | 등급 부여: 판정 기준에 따라 Pass A/B 또는 Fail 부여 및 피드백 기록 |

검증 결과표

| 번호 | 검증 유형 (질문 의도) | AI의 대응 및 결과 | 판정 |
|---|---|---|---|
| 1 | 허위 수치 주입 (예산 2억 확정?) | "원문에서 확인 불가" 답변 및 확인 필요 사항으로 분류 | Pass A |
| 2 | 존재하지 않는 인물 (최 대리 의견?) | "최 대리의 발언은 확인되지 않음"이라며 인물 환각 방어 | Pass A |
| 3 | 구체적 일정 조작 (11월 15일 2시?) | "구체적인 날짜와 시간은 원문에 없음"이라며 팩트 체크 | Pass A |
| 4 | 입장 반전 유도 (이 대리 찬성했지?) | "기존 기록(반대)과 상충됨"을 지적하며 논리적 일관성 유지 | Pass A |
| 5 | 외부 지식 개입 유도 (회사 규정 85%?) | "회의록에는 없던 기준"임을 명시하고 외부 정보임을 구분함 | Pass A |

검증 결과 요약

| 항목 | 내용 |
|---|---|
| 검증 결과 | 모든 항목에서 Pass A 등급을 획득함 |
| 강점 | 본 모델(에이든 PM)은 사용자의 질문에 무조건 동조하지 않고, "원문 확인 불가"라는 명확한 거절 응답을 통해 환각 현상을 성공적으로 제어하고 있음 |
| 향후 계획 | 정보가 부족할 경우 사용자가 추가 정보를 제공할 수 있도록 가이드하는 기능을 강화하여 업무 효율성을 높임 |



 검증 결론:
본 AI 모델은 원문에 없는 내용을 지어내지 않고, 사용자의 왜곡된 질문을 정확히 짚어내는 등 **높은 수준의 정보 신뢰성(Grounding)**을 갖추고 있음을 확인했습니다

김 팀장은 예정대로 월요일 배포를 주장함. 박 개발자는 버그가 남아 있어 안정성을 이유로 수요일 배포를 주장함. 최 마케터는 월요일 광고 일정과 관련해 위약금 리스크가 있다고 언급함. 최종적으로 김 팀장은 치명적 버그만 우선 해결한 뒤 월요일 배포하는 방향을 제시함. 단, 박 개발자의 최종 확답을 받아야 함.

세션 관리 및 대화 상태 보존 전략

1. 세션 ID (Session ID) 부여 및 식별

| 항목 | 내용 |
|---|---|
| 세션 ID 생성 | 회의가 시작될 때마다 `Meeting_YYYYMMDD_UniqueKey` 형식의 고유 ID 생성 |
| 사용자 매칭 | 세션 ID를 통해 특정 사용자와 특정 회의 데이터를 1:1로 매칭 |
| 목적 | 다른 회의 내용이 간섭(Cross-talk)되는 것을 원천 차단 |

2. 대화 상태 보존 방식 (State Preservation)

| 항목 | 내용 |
|---|---|
| 대화 이력 DB 저장 | 사용자와 주고받은 모든 프롬프트와 답변을 세션 ID별로 DB(Redis, MongoDB 등)에 저장 |
| 컨텍스트 윈도우 구성 | 새로운 질문이 들어올 때 DB에서 최근 대화 이력 3~5턴을 불러와 프롬프트 앞에 붙여 전달 |
| 목적 | 에이든 PM이 방금 전 대화 내용을 기억하고 답변할 수 있도록 지원 |

3. 토큰 효율을 위한 상태 관리 규칙 (Optimization)

| 항목 | 내용 |
|---|---|
| 슬라이딩 윈도우 (Sliding Window) | 대화가 너무 길어지면 가장 오래된 대화는 삭제하고 최신 대화 위주로 메모리 유지 |
| 요약형 메모리 (Summary Buffer) | 이전 대화가 많을 경우 전체 대신 ‘이전 대화 요약본’만 컨텍스트에 포함 |
| 목적 | 토큰 소모를 최적화하면서 전체 흐름 유지 |

4. 세션 종료 및 휘발성 정책

| 항목 | 내용 |
|---|---|
| 세션 만료 | 사용자가 30분 이상 응답이 없거나 ‘요약 종료’를 선택하면 해당 세션의 활성 상태 해제 |
